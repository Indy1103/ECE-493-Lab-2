---
command: /speckit.checklist-analyze
description: Perform a checklist-driven requirement quality analysis for the current feature and write one evaluation report markdown file per checklist.
---

## User Input

```text
$ARGUMENTS
```

You **MUST** consider the user input before proceeding (if not empty).

If `$ARGUMENTS` specifies a checklist filename, analyze only that checklist (must exist in the active feature `checklists/` folder).  
If `$ARGUMENTS` is empty, analyze all eligible checklists using the deterministic rule in **Step 2**.

---

## Goal

Analyze checklist file(s) generated by `/speckit.checklist` and determine, for each checklist:

1. Whether each checklist item is **satisfied**, using the specification artifact(s) it explicitly references.
2. If not satisfied, whether the checklist item is **required to exist**, based on:
   - The active Use Case
   - The Acceptance Test Suite
   - The project Constitution

Treat `.specify/memory/constitution.md`, `UseCases.md`, and `TestSuite.md` as the **ground truth** for necessity decisions.  
If a checklist asks for something that is not required by those sources, classify it as **Missing but Not Required**.

At the end of each checklist analysis, produce a **natural-language remediation summary** listing only the **required-but-missing** items in a form suitable for re-running `/speckit.specify`.

Persist the full analysis output to a markdown report file for each analyzed checklist.
Do **not** print full checklist analyses to the command line.

---

## Operating Constraints

### READ-ONLY EXCEPT REPORT OUTPUT
- Do **not** modify specification source files.
- Do **not** rewrite requirements directly.
- Do **not** introduce new requirements.
- Do **not** perform design critique beyond what the checklist item explicitly asks.
- The **only** allowed file write is the checklist evaluation report:
  - `specs/<FEATURE_NAME>/checklists/evaluations/<CHECKLIST_BASENAME>.evaluation.md`

### Authority vs Evidence (MANDATORY)

- **Authority (to decide necessity)**:
  1. `.specify/memory/constitution.md`
  2. `UseCases.md`
  3. `TestSuite.md`

- **Ground-Truth Rule (MANDATORY)**:
  - Authority sources above are the only basis for deciding whether an unsatisfied checklist item is required.
  - Do **not** treat checklist content itself as authority for necessity.
  - If an item is not supported by Constitution/UseCases/TestSuite, it is **not required**.
  - Do **not** recommend editing or extending Constitution/UseCases/TestSuite to satisfy checklist output.

- **Evidence (to check satisfaction)**:
  - Only the file(s) explicitly referenced by the checklist item, such as:
    - `spec.md`
    - `data-model.md`
    - `openapi.yaml`
    - `ux.md`
    - `security.md`
    - `api.md`
  - Other artifacts (e.g., `plan.md`, `research.md`, `quickstart.md`) may be consulted **only if explicitly named by that checklist item**.

`requirements.md` is **explicitly excluded** from evaluation.

---

## Execution Steps

### 1. Establish Feature Context

Determine the active feature directory:

```text
specs/<FEATURE_NAME>/
```

Identify the **active Use Case**:
- Use Case IDs are referenced as `UC-XX`.
- Acceptance Tests are referenced as `AT-UCXX-*`.
- Each feature branch corresponds to **exactly one active Use Case**.

Abort if the active Use Case cannot be identified unambiguously.

---

### 2. Select Checklists to Analyze

If `$ARGUMENTS` specifies a checklist file, use it.

Otherwise, from all checklist files generated by `/speckit.checklist`, select checklists using this deterministic rule:

1. List `specs/<FEATURE_NAME>/checklists/*.md`
2. Exclude `requirements.md`
3. Sort lexicographically by filename
4. Analyze all remaining checklist files in sorted order.

If no eligible checklist exists, report that and stop.

---

### 3. Load Inputs

Load:
- `.specify/memory/constitution.md`
- `UseCases.md`
- `TestSuite.md`
- For each selected checklist file:
  - The checklist file itself
  - Only the specification artifacts explicitly referenced by that checklist’s items

Prepare output location:
- Ensure `specs/<FEATURE_NAME>/checklists/evaluations/` exists.
- For each selected checklist, compute output file:
  - `specs/<FEATURE_NAME>/checklists/evaluations/<CHECKLIST_BASENAME>.evaluation.md`
- Overwrite existing report files with newly generated analysis results.

---

### 4. Checklist Enumeration Guardrail (MANDATORY)

Before evaluating any checklist item, you MUST:

1. Extract **all checklist item IDs** (e.g., `CHK001`, `CHK002`, …) from the checklist.
2. Create an internal ordered list of these IDs in source order.
3. Evaluate checklist items **strictly in ID order**, one by one.
4. Produce output for **every extracted checklist ID**, even if:
   - The item is trivially satisfied
   - The item is not required
   - The item appears redundant

You are NOT allowed to:
- Skip checklist IDs
- Merge multiple checklist items into one
- Reorder checklist items
- Omit an item because it appears “covered” by another

If any checklist ID is missing from the output, analysis is **INVALID**.

---

### 5. Checklist Item Evaluation (PER CHECKLIST)

For **each checklist item (`CHK###`)**:

1. Identify the specification file(s) the item refers to.
2. Determine whether the checklist requirement is **explicitly satisfied**.
3. If satisfied:
   - Mark `[x]`
4. If not satisfied:
   - Determine whether the item is **MANDATED** by:
     - The active Use Case
     - The Acceptance Tests
     - The Constitution
   - If mandate is absent in those authority sources, mark the item **Missing but Not Required**
   - Classify as:
     - **Missing but Required**
     - **Missing but Not Required**

You MUST NOT:
- Infer requirements beyond the checklist text
- Carry findings from other checklists
- Perform cross-checklist analysis unless explicitly requested

---

## CHK-Atomic Output Rule (MANDATORY)

For each checklist item (`CHK###`), you MUST:

1. Emit the checklist line
2. Immediately emit its full evaluation block:
   - Status
   - Required (Yes/No)
   - Authority
   - Evidence Reviewed
3. Complete the block fully BEFORE moving to the next checklist ID

You are NOT allowed to:
- Defer metadata emission
- Group multiple checklist items under a shared metadata block
- Emit authority/evidence blocks without an immediately preceding CHK ID

If any checklist item lacks its own complete block, the analysis is **INVALID**.

---

## Output Format (PER CHECKLIST REPORT)

The following structure is the **report file content** and MUST be written to:
`specs/<FEATURE_NAME>/checklists/evaluations/<CHECKLIST_BASENAME>.evaluation.md`

### Checklist Analysis Result

```md
Output File: specs/<FEATURE_NAME>/checklists/evaluations/<CHECKLIST_BASENAME>.evaluation.md

## Checklist: <Checklist Title>
Target File(s): <file1 | file2 | ...>
```

### Item Blocks (MANDATORY: for **every** checklist item, checked or unchecked)

Use this exact atomic structure:

```md
### CHK###
- [x] CHK### <exact checklist text>      # use [ ] if not satisfied
- Status: Satisfied | Not Satisfied
- Required: Yes | No
- Authority:
  - UC-XX (if applicable)
  - AT-UCXX-YY (if applicable)
  - Constitution §<section> (if applicable)
- Evidence Reviewed:
  - <path> §<section>
### END-CHK###
```

Rules:
- `CHK###` header and `END-CHK###` footer must match.
- Keep one and only one block per ID.
- Do not combine IDs in one block.

---

### Checklist Summary

```md
| Category | Count |
|----------|-------|
| Checklist Items | <N> |
| Satisfied | <S> |
| Missing but Required | <MR> |
| Missing but Not Required | <MNR> |
```

Must satisfy:
- `N = S + MR + MNR`

---

### Required Remediation Summary (Natural Language)

If any items are **Missing but Required**, output:

**You need to add the following to the specification to satisfy this checklist:**

- One bullet per required missing item
- Describe *what must be added*, not *how to implement it*
- Phrase so bullets can be reused as input to `/speckit.specify`
- Do **not** reference checklist IDs
- Do **not** include optional/not-required items

If no required items are missing, output exactly:

> All required checklist items for this checklist are satisfied. No specification updates are required.

---

### Human-Readable Summary (Lab-Ready)

Write one concise paragraph stating:
- Whether this checklist passes
- Whether any required items are missing
- Whether `/speckit.specify` should be re-run and why

---

## Determinism & Validation Gate (MANDATORY)

Before final output, perform these checks:

1. Extracted IDs count equals emitted CHK block count.
2. Emitted IDs exactly match extracted IDs in order.
3. Every CHK block contains all required fields:
   - Status
   - Required
   - Authority
   - Evidence Reviewed
4. Summary arithmetic is correct:
   - `Checklist Items = Satisfied + Missing but Required + Missing but Not Required`

If any check fails, output:

```md
ANALYSIS INVALID: Missing or malformed checklist item blocks.
Missing checklist IDs: <comma-separated IDs or NONE>
Field errors: <brief details>
Re-run required.
```

Write that invalid analysis content to the same checklist report file, then stop.

---

## STOP CONDITION (MANDATORY)

After producing output for all selected checklists:

- Write each checklist's full analysis markdown to:
  - `specs/<FEATURE_NAME>/checklists/evaluations/<CHECKLIST_BASENAME>.evaluation.md`
- Include `Output File: ...` in each saved markdown file so artifacts are self-identifying.
- Command-line output must be minimal and must **not** include CHK item blocks or summary tables.
- Command-line output should include:
  - One line per generated report: `Report written: <path>`
  - Final summary line: `Completed checklist analysis for <N> checklist(s).`
- **STOP**

---

## Non-Goals (EXPLICIT)

This command does **NOT**:
- Evaluate implementation correctness
- Compare different checklists
- Enforce requirements not demanded by the current checklist
- Analyze architectural or design quality

---

## Checklist Completeness Verification (MANDATORY)

At end of valid analysis, append to the saved report file:

```md
The following checklist IDs were detected and evaluated:
CHK001, CHK002, CHK003, … CHK0XX
```

If any source checklist ID is absent from this list, the analysis is **INVALID** and must be rerun.
